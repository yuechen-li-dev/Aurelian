# Aurelian

Aurelian is a greenfield C# engine/runtime project created after the Stri-V rescue effort was paused.

## Status

Aurelian is currently in pre-MVP module bring-up. The repository contains a clean solution skeleton, strict build discipline, architecture notes, smoke tests, the first vendored runtime dependency (Dominatus), and the linked `Aurelian.Shaders`, `Aurelian.Assets`, and `Aurelian.AssetTool` modules.

## What Aurelian is

- A greenfield C# engine/runtime.
- A Dominatus-native runtime spine beginning in A1, with Dominatus vendored under `vendor/Dominatus/`.
- An explicit data-world engine design based on data, composition, logic, and locality of change.
- A typed lifecycle, actuator-owned side-effect, world-to-render-snapshot, command-plan, and graphics-HAL oriented architecture.
- A pragmatic dependency policy: libraries may provide bounded implementation plumbing, but Aurelian owns the architecture spine and hides dependencies behind Aurelian-owned contracts.
- A test-first codebase with nullable reference types and warnings-as-errors for Aurelian-owned projects.
- The home of `Aurelian.Graphics`, the first graphics HAL with Silk.NET Vulkan/windowing references, native-free plant data, and optional/unavailable-safe Vulkan instance/device initialization for plant 0 only.
- The home of `Aurelian.Shaders`, an Aurelian-owned shader module with a new WyrmCoil-inspired SDSL-V AST contract, token-driven parser/validator/HLSL emitter paths, a deterministic SDSL-V artifact manifest M0, optional DXC validation M0, an explicit compatibility matrix, and a temporarily preserved carried-over legacy pipeline.
- The home of `Aurelian.Assets`, an early Aurelian-owned TOML/manifest orchestration module converted from the carried-over Stri-V asset pipeline.
- The home of `Aurelian.AssetTool`, a CLI wrapper over Aurelian asset pipeline functionality converted from the carried-over Stri-V asset tool.

## What Aurelian is not

- It is not a Stride runtime fork.
- It does not use the Stride processor architecture or ECS processors as its runtime core.
- It does not use the Stride asset system as its asset foundation.
- It is not editor-first.
- It does not contain a visual renderer, created window, triangle, Machina integration, WyrmCoil integration, or remaining unconverted Stri-V salvage projects; the only renderer implementation is the headless null backend for command-plan tests, now reachable through world-to-render snapshot extraction. A22 adds windowing package visibility only inside `Aurelian.Graphics`; A23 adds a native-free `PlantId`/`PlantContext`/`PlantRegistry` M0 where `PlantId.Zero` is the single-GPU plant identity and no global graphics singleton is introduced.
- It does not reference or compile WyrmCoil code; WyrmCoil remains reference-only, with SDSL-V semantics compared through an explicit compatibility matrix rather than blindly copied into production code.
- It does not allow libraries to define core world, runtime, actuation, render snapshot, render command plan, render backend, asset, or shader architecture.

## World model doctrine

A12 adds `docs/architecture/world-model-doctrine.md` as the design boundary for future `Aurelian.World` implementation. The doctrine states that locality of change is the primary design law, uses data/composition/logic as the uniform decomposition for worlds, scenes, rooms, NPCs, documents, webpages, UI nodes, shader modules, and asset manifests, and defines an Aurelian component as a reusable local composition unit rather than a passive mutable data bag processed globally.

A13 implements World Unit M0 under `Aurelian.World.Units`: IDs, descriptors, immediate-only composition, a simple resolver, resolved parent/child maps, pre-order traversal, transitive descendant queries, and diagnostics for missing roots, missing children, duplicate immediate children, duplicate child slots, and composition cycles. A14 adds `Aurelian.Actuation.World` as the first typed mutation boundary over those documents: requests are pure data, results always carry a `WorldDocument`, rejected/no-op operations keep the original document, applied operations return a new document, and the resolver validates applied mutations. A16 adds `Aurelian.World.Stores` for explicit typed data stores, not ECS components: `UnitNameStore` attaches names/labels by `UnitId`, `Transform2Store` attaches 2D transforms by `UnitId`, and `WorldDataDocument` groups these stores with the composition document without replacing `WorldDocument`. Store mutation flows through `WorldDataActuator` request/result contracts in `Aurelian.Actuation.World`; invalid names, missing units, and non-finite transforms are rejected with diagnostics, remove-missing operations are no-ops with info diagnostics, and original documents/stores are not mutated in place. A17 adds `Aurelian.Rendering.Contracts.Snapshots` as the renderer-independent DTO boundary for later render production and consumption: snapshots carry frame ids, cameras, items, resource references, statuses, and diagnostics without world, asset, shader, backend, or GPU dependencies. A18 adds command-plan DTOs in the same contracts project, and A19 adds the headless `Aurelian.Rendering.Null` backend that consumes those plans and returns deterministic traces without graphics/windowing dependencies. Future stores may include 3D transform, renderable references, camera references, physics state, or navigation state, but the world project remains library-free and does not reference actuation, Dominatus, rendering, assets, or shaders. M0 deliberately does not introduce a full ECS, processors, an entity manager, behavior runtime, renderer integration, snapshot extraction, asset integration, physics, LLM calls, deep inheritance, or blackboards; blackboards remain a logic/runtime concern rather than durable world data. A15 dependency policy still applies to later actuation/render contract work: use external libraries only behind contracts and keep core behavior reflection-free by default.

## Shader module

A3 links `src/Aurelian.Shaders/Aurelian.Shaders.csproj` into `Aurelian.slnx` with smoke tests under `tests/Aurelian.Shaders.Tests/`. The module was renamed from the carried-over `StriV.ShaderPipeline` project, with namespace and project metadata updated to `Aurelian.Shaders`.

A4 converged the first `Aurelian.Shaders` AST contract toward the WyrmCoil SDSL-V module/declaration/type model in `Aurelian.Shaders.Language.Ast`. A5 added the first token-driven parser M0 path under `Aurelian.Shaders.Language`, including new token, lexer, diagnostic, and parser namespaces. A6 expanded that path to parser M1 statements/expressions: precedence, postfix chains, array literals, with expressions, switch/match parse shapes, fallibility parse shapes, if/else, for, assignment, expression, empty, let, and return statements. A7 adds structural validation M0 under `Aurelian.Shaders.Language.Validation`, with stable validation diagnostics for duplicate declarations/uses/fields/variants/generics/shader members/locals, unknown type references, and invalid array lengths. A8 adds HLSL emission M0 under `Aurelian.Shaders.Language.Emission.Hlsl` over the new parser/AST/validator path, with deterministic output for record/stream structs, shader stage functions, basic statements, and basic expressions. A9 adds artifact manifest M0 under `Aurelian.Shaders.Language.Artifacts`: `SdslvShaderArtifactEmitter` packages source identity, SHA-256 source hash, emitted HLSL, combined parse/validation/emission diagnostics, success state, and heuristic stage/profile entry point metadata into deterministic in-memory artifacts and JSON manifests. A10 adds optional DXC validation M0 under `Aurelian.Shaders.Language.External.Dxc`: `DxcValidator` can validate generated HLSL syntax/profile/entry point combinations one stage at a time when DXC is available, and `DxcDiscovery` checks `AURELIAN_DXC`, then `dxc`, then `dxc.exe` on `PATH`. The checked-in smoke fixture at `tests/Aurelian.Shaders.Tests/Fixtures/Sdslv/smoke_triangle.sdslv` is parsed, validated, emitted, and packaged by xUnit tests so the original SDSL-V source remains inspectable. Validation/emission/artifact/DXC M0 are not full expression type checking, interface satisfaction, SPIR-V generation, renderer/window integration, asset/TOML pipeline integration, or shader behavior evaluation. DXC is external validation only: normal builds/tests do not require DXC, Vulkan SDK is not required, and missing DXC produces skip results rather than failures. Manifest JSON includes HLSL for M0 simplicity and may split HLSL into separate output files later. Stage/profile inference is heuristic in M0. The WyrmCoil ↔ Aurelian compatibility matrix lives in `docs/architecture/sdslv-compatibility-matrix.md`. Aurelian SDSL-V remains its own C# production implementation; WyrmCoil remains reference-only, and old Stride mixins/effect/base-shader compatibility are not native language features. Unsupported HLSL M0 constructs report emission diagnostics instead of using DXC or a renderer. `.sdslvtest` files are future work: Aurelian may eventually support them analogously to `.octest` by porting the Oct interpreter architecture to C# and using a CPU evaluator for deterministic shader behavior tests, but that is intentionally deferred until parser, validation, emission, and artifact contracts are stable.

## A31 graphics upload note

A31 adds the first one-shot staging-buffer to device-local buffer upload path in `Aurelian.Graphics`: CPU bytes are written into a mapped `CpuToGpu` staging buffer, copied with `vkCmdCopyBuffer`, submitted to the plant queue, and synchronized through the command-list timeline fence. The M0 path waits synchronously for completion before disposing staging resources, preserving lifetime safety without adding an upload ring or persistent staging pool.

A32 adds Aurelian-owned barrier/layout planning in `Aurelian.Graphics.Vulkan.Resources.Barriers`: resource layouts, access/stage flags, tested Vulkan mapping facts, pure image barrier plans/batches, a per-subresource layout tracker, and buffer access transition plans. Cross-plant transfer layouts are present as diagnostic stubs for future ownership-transfer work. Command emission, texture resources, render passes, pipelines, descriptors, swapchains/windows/surfaces, and draw paths remain deferred; the recommended next graphics milestone is `A33 — Texture resource M0`.

## A33 Texture2D resource M0

A33 adds the first Vulkan image resource type under `Aurelian.Graphics.Vulkan.Resources.Textures`. Texture creation now uses Aurelian-owned create plans, format and usage enums, creates `VkImage`, queries image memory requirements, allocates only through `IVulkanMemoryAllocator`, binds image memory, optionally creates a default color `VkImageView` for shader-resource/color-attachment use, initializes a per-mip/per-array-layer `VulkanLayoutTracker`, and disposes image view, image, and allocation safely.

M0 intentionally supports only `VulkanResourceLayout.Undefined` as the texture initial layout because Vulkan image creation cannot truthfully start in transfer, shader-resource, or attachment layouts without barrier command emission. Texture upload, copy-to-image commands, image barrier emission, samplers, descriptors, render passes, pipelines, swapchains/windows/surfaces, VMA/VMASharp, and Vortice remain deferred. The recommended next milestone is `A34 — Barrier command emission M1`.

## A34 Barrier command emission M1 note

A34 bridges A32's pure barrier plans into Vulkan command-buffer recording. `Aurelian.Graphics.Vulkan.Resources.Barriers` now has explicit emission request records that pair pure plans with `AurelianVulkanTexture` or `AurelianVulkanBuffer` handles at the backend boundary, plus `VulkanBarrierCommandEmitter` APIs for texture-only, buffer-only, and combined batched `vkCmdPipelineBarrier` recording. Planning remains separate from emission: `VulkanLayoutTracker` still mutates when a transition plan is accepted, and the emitter consumes that plan without mutating tracker state again.

A34 remains deliberately narrow. It records buffer memory barriers and color image layout-transition barriers into an already-recording `VulkanCommandBufferLease`, rejects wrong-plant resources, rejects cross-plant ownership transfer emission, and treats empty barrier lists as successful no-ops so tests pass when Vulkan is unavailable. Texture upload, `vkCmdCopyBufferToImage`, render passes, pipelines, descriptors, swapchains/windows/surfaces, VMA/VMASharp, Vortice, and tracker rollback/reconciliation after emission failure remain deferred.


A35 adds the first whole Texture2D upload path in `Aurelian.Graphics`: CPU RGBA bytes move through a mapped `CpuToGpu` staging buffer, image barriers transition `Undefined -> TransferDestination -> ShaderResourceFragment`, `vkCmdCopyBufferToImage` submits to the plant queue, the command-list timeline fence is signaled and waited synchronously, and staging resources are disposed only after completion. M0 supports whole Texture2D mip0/layer0 four-byte color formats only; mip generation, partial uploads, arrays/cubes, samplers/descriptors, upload rings, render passes, pipelines, draws, swapchains/windows/surfaces, VMA/VMASharp, and Vortice remain deferred.


A36 adds explicit Vulkan render pass descriptor M0 in `Aurelian.Graphics`: render pass attachments are Aurelian-owned plain data with format, load/store ops, and initial/final layouts; the factory compiles the one-color-attachment M0 descriptor to a native `VkRenderPass`; and `AurelianVulkanRenderPass` owns idempotent destruction. Framebuffers, pipelines, draw commands, render-pass begin/end commands, swapchains/windows/surfaces, MSAA, and depth/stencil remain deferred.

A37 adds Vulkan framebuffer M0 in `Aurelian.Graphics`: an explicit `VulkanFramebufferDescriptor` binds exactly one `AurelianVulkanTexture` color attachment to an existing `AurelianVulkanRenderPass`, validates plant/size/usage/image-view/render-pass format compatibility before native calls, and `AurelianVulkanFramebuffer` owns idempotent `VkFramebuffer` disposal. There is no framebuffer cache yet, no command-buffer render pass begin/end, no pipeline/draw path, and no swapchain/window/surface, depth/stencil, MSAA, descriptor-set, VMA/VMASharp, or Vortice work.

## A43 compiled shader contract bridge note

A43 adds the neutral compiled shader stage contract bridge between the shader artifact path and Vulkan pipeline descriptors. `Aurelian.Rendering.Contracts.Shaders` now owns renderer- and compiler-independent DTOs for compiled shader programs, stages, statuses, and diagnostics. `Aurelian.Shaders` exports A41/A42 SPIR-V artifacts to those neutral contracts, while `Aurelian.Graphics` maps neutral compiled stages to Vulkan graphics shader stage descriptors by validating byte lengths, SPIR-V magic, graphics-supported stages, and little-endian word conversion.

The dependency boundary remains intentional: `Aurelian.Shaders` and `Aurelian.Graphics` do not reference each other. DXC remains scoped to shader tooling/artifact production, and graphics consumes compiled contract data only. Pipeline artifact consumption remains staged; A43 does not add runtime shader compilation, asset/TOML integration, SDSL-V-to-pipeline creation, draw commands, descriptor sets, swapchains, windows, or surfaces.
